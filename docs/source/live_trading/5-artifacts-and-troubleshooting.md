# 产物清单与排错

本页提供 live 运行时最常用的排错路径与运维建议。

## 0. 适用场景

- 你已经在 live 模式运行，但遇到“提交失败 / 拒单 / 状态异常理解”等问题
- 你希望按固定顺序快速定位问题来源，减少盲目排查

## 1. 关键产物与用途

- `sys_log`：系统运行与关键流程日志
- `trade_log`：交易明细及相关输出
- `risk_log`：风控拒绝与风险相关事件
- 断点文件：用于恢复运行状态

建议把三类日志分别用于三类问题：

- 运行是否正常：优先看 `sys_log`
- 下单为何被拒：优先看 `risk_log`
- 状态为何变化：优先看 `trade_log`

## 2. 建议排错顺序

1. 先看即时界面反馈（CLI/TUI）  
2. 再看 `sys_log` 的错误与警告  
3. 若是拒单，检查 `risk_log` 的 `rule_id` / `reason`  
4. 最后对照订单与成交记录确认状态是否符合预期  

如果是“看起来没报错但行为不对”，建议再补一步：

5. 回看本次运行配置快照（账户、资产类型、broker、频率）

## 3. 高频故障剧本

### A. 订单被拒绝

- 现象：下单后提示拒绝
- 排查：定位对应 `rule_id` 与英文 `reason`
- 处理：调整规则阈值或订单参数，复测同一场景

### B. 提交失败

- 现象：未产生有效提交记录
- 排查：检查连接状态、契约字段完整性
- 处理：先修复连接/字段问题，再重试并对照 `sys_log`

### C. 状态理解偏差

- 现象：订单长期 `partial-filled`
- 排查：确认累计成交量与目标量关系
- 处理：结合成交回报判断是否应转为 `filled`

### D. 收盘后订单处理疑问

- 现象：收盘后订单状态变化不确定
- 排查：查看收盘后处理日志与撤单记录
- 处理：按收盘处理策略核对最终状态

### E. 界面与日志不一致

- 现象：CLI/TUI 提示与日志感知不一致
- 排查：按时间顺序对齐界面反馈和日志条目
- 处理：以日志中的最终状态为准，复核同一订单 ID 的全链路记录

## 4. 运维建议

- 定期检查日志目录容量与保留策略
- 保留至少一段可复盘窗口，便于问题复现
- 对关键运行日保存配置快照与日志头信息

## 5. 快速检查清单

- 账户是否正确
- 配置是否生效
- 风控是否拒绝
- broker 是否连通
- 回报是否满足契约
- 日志是否完整可读

## 6. 常用英文提示样例

```text
Order rejected by risk rule [MAX_ORDER_QTY]: order quantity exceeds limit
```

```text
Order submission failed.
```
